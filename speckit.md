## التشغيل
1. افتح `index.html` بمتصفح حديث (Chrome/Edge/Firefox) — **ليس** وضع التصفّح الخاص.
2. أول تشغيل يولّد بيانات تجريبية تلقائيًا.

## الحسابات التجريبية
| الدور | المستخدم | كلمة المرور |
|---|---|---|
| مدير الموقع | `admin` | `admin123` |
| عضو لجنة | `member` | `member123` |

- من وحدة تحكّم المتصفح:


# الدستور — مبادئ مشروع عائلة النجار (Constitution)

هذه المبادئ **غير قابلة للتفاوض**. أي مخرَج يخالف مبدأً هنا يُرفض ويُعاد. كل مبدأ قابل للاختبار.

---

## المادة 1 — صفر اعتماديات خارجية (المرحلة 1)
- ممنوع أي إطار/مكتبة JS أو CSS (لا React، لا jQuery، لا Bootstrap، لا Chart.js…).
- ممنوع أي طلب شبكة وقت التشغيل عدا خط الخطوط (Google Fonts) وهو **اختياري** مع بديل نظام.
- الرسوم البيانية تُرسم يدويًا بـ SVG داخل الكود.
- **اختبار:** فتح الملف بدون إنترنت يعمل بالكامل (عدا شكل الخط).

## المادة 2 — عزل طبقة البيانات
- كل قراءة/كتابة تمرّ عبر واجهة `Store` واحدة (`open/all/get/put/del/clear`) — راجع `contracts/data-layer-api.md`.
- ممنوع استدعاء `indexedDB` مباشرة من دوال العرض؛ فقط من داخل طبقة `Store`.
- **الهدف:** استبدال IndexedDB بـ RealmDB في المرحلة 2 دون لمس دوال الواجهة.
- **اختبار:** البحث عن `indexedDB` يجب أن يظهر فقط داخل تعريف `Store`.

## المادة 3 — العربية أولًا وإتاحة الوصول
- الاتجاه RTL، اللغة عربية في كل نص واجهة.
- تباين لوني كافٍ (AA)، تركيز لوحة مفاتيح ظاهر (`:focus-visible`)، احترام `prefers-reduced-motion`.
- **اختبار:** التنقّل بالكيبورد ممكن، والحركة تتوقّف عند تفعيل تقليل الحركة.

## المادة 4 — دلالات الألوان ثابتة
- الأزرق: البنية والتفاعل الأساسي. الأخضر: التبرعات والإيجابي والنجاح. الأحمر: التنبيه والحذف والخطر.
- **اختبار:** زر الحذف أحمر دائمًا، وقيم التبرعات خضراء.

## المادة 5 — صدق أمني

- **اختبار:** الوثائق تذكر هذا القيد صراحة.

## المادة 6 — كل ميزة لها معيار قبول
- لا تُعتبر مهمة منجزة إلا بتحقّق «معيار القبول» الخاص بها في `tasks.md`.
- **اختبار:** كل مهمة في `tasks.md` تحتوي حقل «القبول».

## المادة 7 — مخرَج المرحلة 1: ملف واحد

# عقد طبقة البيانات — `contracts/data-layer-api.md`
وقّعات ثابتة. نفّذها كما هي. هي الحدّ الفاصل بين المنطق والتخزين (يُستبدل بـ Realm لاحقًا دون تغيير المستدعين).

---

## الكائن `DB` (طبقة `Store`)
```js
const DB = {
  // يفتح/يهيّئ قاعدة البيانات وينشئ كل المخازن إن لزم.
  open(): Promise<void>,

  // يعيد كل السجلات في مخزن كمصفوفة.
  all(store: string): Promise<object[]>,

  // يعيد سجلًا بمعرّفه أو undefined.
  get(store: string, id: string): Promise<object|undefined>,

  // يضيف أو يحدّث سجلًا. إن لم يكن له id يولّد واحدًا. يعيد السجل المحفوظ.
  put(store: string, value: object): Promise<object>,

  // يحذف سجلًا بمعرّفه. يعيد true.
  del(store: string, id: string): Promise<true>,

  // يمسح كل سجلات مخزن. يعيد true.
  clear(store: string): Promise<true>,

  // ثابت: أسماء كل المخازن.
  STORES: string[]
}
```

### شروط السلوك (Contracts)
1. `open()` ينشئ كل مخزن في `STORES` بـ `keyPath:"id"` إن لم يوجد. اسم القاعدة `najjar_family_db`، الإصدار `1`.
2. `put` يضبط `value.id = uid()` إذا كان غائبًا، ثم يخزّن، ثم يعيد `value` نفسه.
3. كل الدوال ترجع Promise؛ الأخطاء تُرفض (reject) لتُلتقط في `boot()`.
4. لا منطق أعمال داخل `Store` (لا تحقّق حقول، لا صلاحيات) — فقط تخزين خام.
5. **قاعدة الدستور:** لا يُستدعى `indexedDB` خارج هذا الكائن إطلاقًا.

### مساعد إلزامي
```js
function uid(): string   // Date.now().toString(36) + random(5) — معرّف فريد نصّي
```

---

## عقد المرحلة 2 (RealmDB) — نفس الوقّعات
عند الانتقال للموبايل، يُعاد تنفيذ `DB` فوق Realm مع **الحفاظ على نفس الأسماء والوقّعات**:
```js
DB.open()  → فتح Realm({schema})
DB.all(s)  → realm.objects(s) → مصفوفة
DB.get     → realm.objectForPrimaryKey(s,id)
DB.put     → realm.write(()=> realm.create(s,value,'modified'))
DB.del     → realm.write(()=> realm.delete(obj))
DB.clear   → realm.write(()=> realm.delete(realm.objects(s)))
```
الطبقات 3–9 (State/Auth/Views…) **لا تتغيّر** لأنها تعتمد على هذا العقد فقط.

## قواعد التحقّق (Validation)
- الحقول الإلزامية أعلاه لا تُحفظ فارغة (تحقّق نموذج المتصفح `required`).
- `amount`, `target`, `total`, `beneficiaries[].amount` أرقام ≥ 0.
- `role` ∈ {admin, member}. `type`(branch) ∈ {فخذ, فرع}.
- التواريخ تُخزَّن ISO `YYYY-MM-DD` وتُعرض بصيغة عربية طويلة.
- `total` في `distributions` **يُحسب دائمًا** من المستفيدين، لا يُدخل يدويًا.

# سجل القرارات التقنية — `research.md`

### R-01 التخزين المحلي
- **البدائل:** localStorage، IndexedDB، بالذاكرة فقط.

- **المبرّر:** مخازن كائنية منظّمة وسعة أكبر، وهي الأقرب مفاهيميًا لـ RealmDB ما يسهّل الانتقال للمرحلة 2. localStorage مفتاح/قيمة محدود السعة؛ الذاكرة تفقد البيانات.

### R-02 لا أطر خارجية
- **البدائل:** React/Vue، Bootstrap/Tailwind، Chart.js.
- **القرار:** فانيلا JS/CSS + SVG يدوي.
- **المبرّر:** طلب صريح، استقلالية تامة (يعمل بلا إنترنت)، حجم أصغر، وتحكّم كامل. الكلفة: كتابة رسوم SVG يدويًا — مقبولة لثلاثة أنواع فقط.

### R-03 SPA بملف واحد
- **البدائل:** متعدد الصفحات، متعدد الملفات.
- **القرار:** SPA في `index.html` واحد مع مُوجّه بسيط.
- **المبرّر:** مشاركة وتشغيل أسهل (نقرة واحدة)، حالة موحّدة، لا إعادة تحميل. الصيانة تُدار بفواصل أقسام واضحة.

### R-04 عزل طبقة البيانات
- **القرار:** كائن `Store` وحيد يغلّف IndexedDB.
- **المبرّر:** المرحلة 2 تستبدل التطبيق الداخلي بـ Realm دون لمس المنطق أو العرض. مبدأ Ports & Adapters مبسّط.

### R-05 نسخة ذاكرة `CACHE`
- **القرار:** تحميل كل المخازن إلى `CACHE` وتحديثها بعد كل كتابة.
- **المبرّر:** IndexedDB غير متزامن؛ العرض المتزامن من `CACHE` أبسط وأسرع ويجنّب await داخل قوالب العرض.

### R-06 رسوم SVG يدوية
- **القرار:** donut (أقواس stroke-dasharray)، bar (مستطيلات)، line (polyline + polygon مساحة).
- **المبرّر:** كافية للوحة، بلا اعتماديات، متجاوبة عبر viewBox.

### R-07 مصادقة عميل (نموذج)
- **القرار:** مطابقة username/password من مخزن users على العميل.
- **المبرّر:** يكفي للنموذج والعرض. موثّق صراحة أنه غير آمن؛ الأمان الحقيقي مرحلة خادم.

### R-08 معرّفات عشوائية نصّية
- **القرار:** `uid() = Date.now(36)+random`.
- **المبرّر:** تجنّب تضارب المفاتيح دون عدّاد مركزي، ومناسب لبيئة لا‑خادمية.

### R-09 استمرار الجلسة عبر `meta`
- **القرار:** تخزين `{id:'session',userId}` في مخزن meta.
- **المبرّر:** إعادة التحميل لا تطرد المستخدم؛ تجربة أفضل دون نظام جلسات معقّد.

### R-10 مستفيدو التوزيع كنص

## 1. نظرة عامة
نظام داخلي للجنة عائلة النجار لإدارة: الأفخاذ والفروع، أفراد العائلة، المبادرين وتبرعاتهم، أصناف مشاريع التبرع، جداول التوزيع، والمناسبات — مع لوحة إحصائية وتقارير وإشعارات، وتخزين محلي.

### الأهداف
- توحيد بيانات العائلة في نظام واحد منظّم.
- شفافية التبرعات (من تبرّع، لأي مشروع، وكم) ومتابعة الإنجاز مقابل المستهدف.
- تسهيل التوزيع العادل وتوثيقه.
- إبقاء اللجنة على اطّلاع عبر لوحة تحكم وإشعارات.

### خارج النطاق (Non-Goals) — المرحلة 1
- الدفع الإلكتروني، الرسائل الجماعية (SMS/WhatsApp)، المزامنة السحابية، تطبيق الموبايل، المصادقة الآمنة الحقيقية. (كلها مراحل لاحقة.)

## 2. الأدوار (Personas)
- **مدير الموقع (admin):** صلاحية كاملة على كل الأقسام + إدارة المستخدمين + النسخ الاحتياطي والاستعادة + سجل النشاطات.
- **عضو لجنة العائلة (member):** إضافة/تعديل/حذف بيانات التشغيل (أفراد، مبادرون، تبرعات، أصناف، توزيع، مناسبات) + عرض اللوحة والتقارير والإشعارات وسجل النشاطات. **بدون** إدارة مستخدمين ولا استعادة نسخ احتياطية.

## 3. قصص المستخدم ومعايير القبول

> صيغة القبول: **إعطاء** (السياق) / **عند** (الفعل) / **إذًا** (النتيجة المتوقّعة).

### US-01 تسجيل الدخول
كعضو، أريد الدخول باسم مستخدم وكلمة مرور لأصل للنظام حسب دوري.
- **AC-01.1:** إعطاء بيانات صحيحة، عند الضغط دخول، إذًا يفتح النظام وتظهر أقسام دوري فقط.
- **AC-01.2:** إعطاء بيانات خاطئة، عند الدخول، إذًا تظهر رسالة خطأ واضحة ولا يدخل.
- **AC-01.3:** إعطاء مستخدم «member»، إذًا لا يظهر قسما «إدارة المستخدمين» و«النسخ الاحتياطي».

### US-02 الأفخاذ والفروع
كعضو، أريد إدارة الأفخاذ وفروعها لتنظيم بنية العائلة.
- **AC-02.1:** أستطيع إضافة/تعديل/حذف فخذ (اسم، رئيس، ملاحظات).
- **AC-02.2:** أستطيع إضافة فرع مرتبط بفخذ أب.
- **AC-02.3:** يظهر لكل فخذ/فرع عدد الأفراد المنتمين إليه، بعرض شجري.

### US-03 أفراد العائلة وشجرتها
كعضو، أريد سجل أفراد وشجرة عائلة لمعرفة العلاقات.
- **AC-03.1:** بيانات الفرد: الاسم، الفخذ/الفرع، الجنس، تاريخ الميلاد، الهاتف، الحالة (حي/متوفى)، الأب، ملاحظات.
- **AC-03.2:** أستطيع التبديل بين عرض «قائمة» و«شجرة» مبنية على علاقة الأب–الابن.
- **AC-03.3:** أستطيع البحث والتصفية حسب الفخذ والحالة.
- **AC-03.4:** الفرد المتوفى يظهر بتمييز بصري.

### US-04 أصناف مشاريع التبرع
كعضو، أريد تعريف أصناف المشاريع لأربط التبرعات بها وأتابع الإنجاز.
- **AC-04.1:** أستطيع إضافة/تعديل/حذف صنف (اسم، وصف، مبلغ مستهدف، لون).
- **AC-04.2:** يظهر لكل صنف: المُجمّع مقابل المستهدف ونسبة مئوية وشريط تقدّم.

### US-05 المبادرون
كعضو، أريد سجل المبادرين لأربط تبرعاتهم بهم.
- **AC-05.1:** أستطيع إضافة/تعديل/حذف مبادر (اسم، هاتف، فخذ، ملاحظات).
- **AC-05.2:** يظهر لكل مبادر إجمالي تبرعاته وعددها، مرتّبين تنازليًا حسب الإجمالي.

### US-06 التبرعات
كعضو، أريد تسجيل التبرعات وربطها بمبادر وصنف.
- **AC-06.1:** أستطيع تسجيل تبرع (المبادر، الصنف، المبلغ، التاريخ، طريقة الدفع، ملاحظات).
- **AC-06.2:** لا أستطيع تسجيل تبرع قبل وجود مبادر وصنف (رسالة إرشادية).
- **AC-06.3:** عند تبرع جديد يُنشأ إشعار تلقائي.
- **AC-06.4:** أستطيع تصدير التبرعات إلى CSV.

### US-07 جداول التوزيع
كعضو، أريد إنشاء جداول توزيع مع المستفيدين.
- **AC-07.1:** أستطيع إنشاء جدول (عنوان، صنف، تاريخ، حالة: مخطط/جارٍ/مكتمل) وإضافة مستفيدين (اسم + مبلغ).
- **AC-07.2:** يُحسب الإجمالي تلقائيًا من مجموع مبالغ المستفيدين.

### US-08 المناسبات والتقويم
كعضو، أريد تسجيل المناسبات لأتابع القادم منها.
- **AC-08.1:** أستطيع إضافة مناسبة (عنوان، نوع: فرح/وفاة/عزاء/اجتماع، تاريخ، مكان، وصف).
- **AC-08.2:** تُميَّز المناسبات القادمة، وتظهر أقربها في لوحة التحكم.

### US-09 لوحة التحكم
كعضو، أريد نظرة عامة سريعة.
- **AC-09.1:** بطاقات مؤشرات: عدد الأفخاذ، الفروع، الأفراد، المبادرين، إجمالي التبرعات.
- **AC-09.2:** ثلاثة رسوم: التبرعات عبر آخر 6 أشهر، التبرعات حسب الصنف، توزيع الأفراد على الأفخاذ.
- **AC-09.3:** قائمتا: أحدث النشاطات، والمناسبات القادمة.

### US-10 الإشعارات
كعضو، أريد رؤية التنبيهات.
- **AC-10.1:** جرس بعدّاد غير المقروء، ولوحة منسدلة بأحدث الإشعارات.
- **AC-10.2:** أستطيع تعليم الكل كمقروء ومسح الكل.

### US-11 التقارير والطباعة
كعضو، أريد تقارير قابلة للطباعة والتصدير.
- **AC-11.1:** لكل من (تبرعات، مبادرون، أفراد، توزيع، أصناف، مناسبات): طباعة (PDF عبر المتصفح) وتصدير CSV.
- **AC-11.2:** تقرير الطباعة يحمل ترويسة العائلة والتاريخ وعدد السجلات.

### US-12 سجل النشاطات
كعضو، أريد تتبّع من فعل ماذا ومتى.
- **AC-12.1:** كل إضافة/تعديل/حذف تُسجَّل باسم المستخدم والعنصر والوقت.

### US-13 النسخ الاحتياطي (مدير فقط)
كمدير، أريد حماية البيانات.
- **AC-13.1:** أستطيع تصدير كامل قاعدة البيانات كملف JSON.
- **AC-13.2:** أستطيع الاستعادة من ملف (بعد تأكيد) فتُستبدل كل البيانات.

### US-14 إدارة المستخدمين (مدير فقط)
كمدير، أريد إدارة حسابات اللجنة.
- **AC-14.1:** أستطيع إضافة/تعديل/تعطيل مستخدم وتحديد دوره.
- **AC-14.2:** لا أستطيع حذف حسابي الحالي.

# عقد مكوّنات الواجهة — `contracts/ui-behaviors.md`
مكوّنات مشتركة بوقّعات ثابتة تُبنى مرّة وتُعاد في كل قسم لضمان الاتساق.

---

## 1. التوست (إشعار لحظي)
```js
function toast(msg: string, type?: "ok"|"err"|"info"): void
```
- يظهر أسفل الشاشة، يختفي بعد ~2.6s. `ok`=أخضر، `err`=أحمر، `info`=أزرق.

## 2. المودال العام
```js
function openModal(title: string, bodyHTML: string, footHTML?: string): void
function closeModal(): void
```
- خلفية معتمة، الإغلاق بالنقر خارجه أو زر ✕ أو مفتاح Escape.

## 3. نافذة التأكيد (قبل الحذف/الاستعادة)
```js
function confirmDialog(msg: string, onYes: () => Promise<void>|void): void
```
- زر «نعم، احذف» أحمر يستدعي `onYes` ثم يغلق. إلزامي قبل أي فعل مدمّر (المادة 8).

## 4. النموذج المُولّد (المحرّك الأساسي لكل CRUD)
```js
function formModal(title, fields, values, onSubmit): void
```
- `fields: Array<{ name, label, type?, options?, required?, col?, rows?, ph? }>`
  - `type`: `"text"|"number"|"date"|"tel"|"select"|"textarea"` (افتراضي text).
  - `options`: للـ select — مصفوفة نصوص أو `{value,label}`.
  - `col:"full"` يمدّ الحقل على العمودين.
- `values`: قيم أولية للتعديل (أو null للإضافة).
- `onSubmit(obj)`: يُستدعى بعد التحقّق؛ `obj` يجمع القيم (number يُحوّل رقمًا، نص يُشذّب).
- يتحقّق من `required` عبر `reportValidity()` قبل الإرسال.

**مثال استخدام (تبرع):**
```js
formModal('تسجيل تبرع', [
  {name:'donorId',   label:'المبادر', type:'select', required:true, options:donorOpts},
  {name:'categoryId',label:'الصنف',  type:'select', required:true, options:catOpts},
  {name:'amount',    label:'المبلغ (₪)', type:'number', required:true},
  {name:'date',      label:'التاريخ', type:'date', required:true},
], null, async (o)=>{ await DB.put('donations', o); /* ...النمط الإلزامي... */ });
```

## 5. الرسوم البيانية (SVG)
```js
function donutChart(data: {label,value,color}[]): string   // يعيد HTML (svg + legend)
function barChart(data: {label,value,color?}[]): string     // أعمدة عمودية
function lineChart(points: {label,value}[]): string         // خط + مساحة
```
- كلها تعيد نصّ HTML يُدرج مباشرة. تتعامل مع القوائم الفارغة بأمان (رسالة «لا بيانات»).

## 6. الحالة الفارغة
```js
function emptyState(icon: string, title: string, subtitle: string): string
```
- تُعرض بدل الجداول/الشبكات الفارغة.

## 7. الشارات (Pills) — أصناف CSS
```
.pill.blue  .pill.green  .pill.red  .pill.gray  .pill.gold
```
- تُستخدم للحالات: نشط=green، معطّل=gray، مدير=red، مكتمل=green، جارٍ=blue، مخطط=gold.

## 8. التنقّل والصلاحيات
```js
const SECTIONS = [{group, items:[{id,label,icon,store?,admin?}]}]
function renderNav(): void        // يبني القائمة ويُخفي عناصر admin عن member
function navigate(route): void     // يبدّل القسم ويستدعي view<Section>()
function allowed(item): boolean    // false إذا item.admin و المستخدم ليس admin
```

## 9. الجرس والإشعارات
```js
function renderBell(): void        // يحدّث عدّاد غير المقروء ولوحة أحدث 8 إشعارات
function markAllRead(): void
function clearNotifs(): void
```

## 10. تنسيقات إلزامية (مساعدات عرض)
```js
esc(s)      // تهريب HTML — إلزامي حول أي قيمة مستخدم داخل القوالب
money(n)    // "n ₪" بفواصل آلاف عربية
num(n)      // رقم بفواصل عربية
fmtDate(d)  // تاريخ عربي طويل أو "—"
relTime(ts) // "قبل س/د/يوم"
```
> **أمان:** لفّ كل قيمة قادمة من المستخدم بـ `esc()` داخل قوالب القوالب النصّية (منع XSS).
